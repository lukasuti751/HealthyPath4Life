About HealthyPath4Life

HealthyPath4Life is the web interface for the Veg3n on-chain companion. Veg3n lets you log meals (as hashes), set nutrition goals, join time-bound paths and earn points. The path coordinator creates paths; the reward keeper awards points. All data lives on the EVM chain.

The companion is designed for an AI healthy eating lifestyle: you record what you eat (as a hash for privacy or consistency), tag meals with path tags (e.g. balanced, plant-based), and optionally join community paths. Points and daily streaks encourage consistency. Goals let you set a target (stored as a hash and value) for your nutrition journey.

This site does not store your meal data on a server; hashing is done in your wallet or in the v3gn CLI. The contract stores only hashes and metadata on-chain.

Meal logging

Log a meal with a meal hash, path tag and type (breakfast, lunch, dinner, snack). Optionally batch log.

Paths

Paths are time-bound (start/end block). Join a path to track your journey with others.

Points & goals

Earn points when the reward keeper awards them. Set a goal (hash + target value) for your nutrition target.

Paths & tips

Paths are created by the path coordinator. You can join or leave while a path is active. Tips (AI healthy eating tips) are recorded as hashes by the coordinator.

Meal types: 1 = Breakfast, 2 = Lunch, 3 = Dinner, 4 = Snack. Hash your meal data off-chain (e.g. keccak256) before logging.

Meal types reference

  • 1 — Breakfast
  • 2 — Lunch
  • 3 — Dinner
  • 4 — Snack

Breakfast (1)

Log morning meals: oatmeal, eggs, fruit, smoothies, etc. Use mealHash = keccak256 of your meal description.

Lunch (2)

Midday meals. Combine with a path tag (e.g. "balanced", "plant-based") for tracking.

Dinner (3)

Evening meals. Same hashing and path tag logic as other types.

Snack (4)

Snacks between meals. Helps build a full picture of your eating pattern.

Path lifecycle

The path coordinator creates a path with createPath(pathTag, startBlock, endBlock). Paths are identified by a sequential pathId. While the current block is between startBlock and endBlock, the path is active. Users call joinPath(pathId) to participate and leavePath(pathId) to leave. Path participant count is updated automatically. Use getActivePathIds() to list currently active paths.

FAQ

        

          
What is Veg3n?

          
Veg3n is an on-chain AI healthy eating companion. You log meals (as hashes), set goals, join paths and earn points. Path coordinator and reward keeper are set at deployment.

          
How do I log a meal?

          
Call logMeal(mealHash, pathTag, mealType). Hash your meal description off-chain (e.g. keccak256). Meal type: 1=breakfast, 2=lunch, 3=dinner, 4=snack.

          
What are paths?

          
Paths are time-bound (start block, end block). The path coordinator creates them. You join with joinPath(pathId) while the path is active.

          
How do I get points?

          
The reward keeper awards points via awardPoints or awardPointsForMeal. You can redeem points with redeemPoints(amount, reasonHash).

          
What are daily snapshots?

          
The reward keeper can record daily snapshots (user, day block, meal count, points earned) for analytics.

          
What is the daily streak?

          
The contract updates a per-user streak when you log meals. It increments if you log again within a short block window; otherwise it resets to 1.

          
How do I set a goal?

          
Call setGoal(goalHash, targetValue). The goal hash could represent a nutrition target (e.g. calories, protein); target value is a number. Only one goal per user (updated on each set).

          
Can I remove a meal?

          
Yes. As the meal owner you can call removeMeal(mealId). The reward keeper can also remove any meal.

          
What are path tag labels?

          
The path coordinator can set a label hash for a path tag via setPathTagLabel(pathTag, labelHash). Useful for off-chain display of category names.

          
Is the contract pausable?

          
Yes. The owner can set companionPaused. When paused, meal logging, path creation, tip recording and similar actions are disabled.

Daily streak & goals

        
The Veg3n companion tracks a daily streak per user. Each time you log a meal, the contract updates lastLogBlock and may increment dailyStreak if the block distance is within a threshold (roughly same “day”). Use getDailyStreak(user) and getLastLogBlock(user) to read. Goals are stored per user: goalHash (bytes32) and targetValue (uint256). Set with setGoal(goalHash, targetValue). Read with getUserGoal(user) or getGoalHash / getGoalTargetValue / getGoalSetAtBlock.

      

    


    

      

        
Glossary

    •

    •

    • 
          
  • Daily snapshot — Optional record: user, dayBlock, mealCount, pointsEarned.
    • 
          
  • Companion domain — keccak256-derived identity for the contract at deployment.

How to hash for logging

        
Meal and path tag are stored as bytes32. Hash off-chain (e.g. with ethers.js):

        
// ethers v6
import { ethers } from "ethers";
const mealHash = ethers.keccak256(ethers.toUtf8Bytes("Oatmeal with berries"));
const pathTag = ethers.keccak256(ethers.toUtf8Bytes("balanced"));
// Then: contract.logMeal(mealHash, pathTag, 1); // 1 = breakfast

Deploy & configure

        
Compile Veg3n with Solidity 0.8.24. Use OpenZeppelin ReentrancyGuard, Pausable, Ownable. The constructor takes no arguments; pathCoordinator, rewardVault and rewardKeeper are set inside the contract (immutable). After deployment, the owner can set points per meal and pause state. The path coordinator should create paths and optionally set path tag labels and record tips. The reward keeper can award points and record daily snapshots.

      

    


    

      

        
Integration guide

        
    
          
  1. Deploy the Veg3n contract (no constructor args). Note the contract address.
    2. 
          
  2. Configure the path coordinator, reward vault and reward keeper addresses in your backend if needed (they are already in the contract).
    3. 
          
  3. In the HealthyPath4Life UI, set the contract address in the Config tab. It is saved in localStorage.
    4. 
          
  4. Connect a wallet (e.g. MetaMask) that holds gas on the target chain.
    5. 
          
  5. To log a meal: compute mealHash = keccak256(mealDescription) and pathTag = keccak256(tagString). Call logMeal(mealHash, pathTag, mealType) with mealType 1–4.
    6. 
          
  6. To join a path: the path coordinator must have created a path with createPath(pathTag, startBlock, endBlock). Users call joinPath(pathId) when the path is active.
    7. 
          
  7. To award points: the reward keeper calls awardPoints(user, amount) or awardPointsForMeal(mealId).
    8. 
          
  8. To set a goal: any user calls setGoal(goalHash, targetValue). One goal per user; each call overwrites.
    9. 
          
  9. To redeem points: user calls redeemPoints(amount, reasonHash).
    10. 
          
  10. To record a tip: path coordinator calls recordTip(tipHash).
    11. 
          
  11. To record a daily snapshot: reward keeper calls recordDailySnapshot(user, dayBlock, mealCount, pointsEarned).
    12. 
          
  12. Read digest, meals, paths, points and other state via the Read tab or direct contract calls.

Chain support

        
Veg3n is chain-agnostic. Deploy on Ethereum mainnet, Arbitrum, Optimism, Base, Polygon, or any EVM-compatible network. Path coordinator, reward vault and reward keeper are 40-hex EIP-55 style addresses set in the constructor. Ensure your wallet and RPC point to the same chain as the deployed contract.

      

    


    

      

        
Troubleshooting

        
    
          
  • Wallet not connecting: Ensure you have a browser wallet (e.g. MetaMask) and that the site is allowed to access it. Try refreshing and reconnecting.
    • 
          
  • Contract address invalid: Paste the full 0x + 40 hex address of your deployed Veg3n contract. Check the chain (mainnet vs testnet).
    • 
          
  • logMeal fails: Ensure mealHash and pathTag are bytes32 (0x + 64 hex). Meal type must be 1, 2, 3 or 4. Contract must not be paused.
    • 
          
  • joinPath fails: Path must exist and current block must be between path start and end. You must not already be on that path.
    • 
          
  • Insufficient points: redeemPoints requires your points balance to be at least the amount. Only the reward keeper can award points.
    • 
          
  • Get digest / read fails: Confirm contract address and chain. Some view functions revert if the ID is out of range (e.g. mealId 0 or beyond mealCounter).
    • 
          
  • Batch log: Use batchLogMeals with arrays of equal length; max length is V3G_MAX_BATCH_MEALS (42).
    • 
          
  • Path not active: Check block numbers. getPath(pathId) returns startBlock and endBlock; current block must be in [startBlock, endBlock].
    •
